# 🤝 開發與協作規範指南 (Development & Collaboration Guide)

為了保持本專案代碼庫的整潔與歷史紀錄的可讀性，請所有開發人員在提交任何修改前，遵循以下協作規範與 Pull Request (PR) 流程。

---

## 1. 協作模型：先「分叉」再修改
本專案採用 **「分叉合併模型 (Fork and Pull Model)」** 來進行團隊協作：
* **優點**： 讓開發人員在自己的倉庫副本中放心進行開發、測試與修正，避免直接對主倉庫造成非預期的影響。
* **操作**： 點擊主倉庫右上角的 "Fork" 按鈕，建立您個人帳號下的倉庫副本。

## 2. 開發流程：建立「主題分支」
嚴禁直接在 `main`（主分支）上進行修改與提交。請為每個新功能或 Bug 修正建立一個獨立的「主題分支」（例如: `feature-ui-update` 或 `bugfix-auth`）。
* **為什麼**： 保持開發範疇單一，若同時進行多個任務時代碼不會混雜。
* **操作**：
  1. 將您 Fork 後的個人倉庫 Clone 到本機。
  2. 建立並切換至新分支：`git checkout -b 分支名稱`。

## 3. 關於「壓扁合併 (Squash)」：不用擔心 Commit 太多
本專案在合併 PR 時，統一採用 **「壓扁合併 (Squash and Merge)」** 併入 `main`：
* **機制**： 不論您在開發主題分支時提交了多少次 commits（例如多次的微調與測試紀錄），在合併時都會被打包壓縮成 **一個乾淨的正式 Commit 紀錄**。
* **好處**： 開發過程中您可以隨時 commit 來記錄進度，不用刻意整理 commit 歷史。管理員會在合併時幫團隊把關歷史整潔。

## 4. 提交 PR 的規範
當功能開發與測試完成，準備合併回主專案時，請發起 Pull Request (PR)：
* **標題規範**： 請使用一句話精準描述您所做的修改（此標題將成為 Squash 合併後的正式 commit 訊息，例如：`修正訂單編輯頁面的金額四捨五入邏輯`）。
* **說明內容**： 請依 PR 範本填寫變更描述、調整範疇與自我檢查清單。
* **等待審查 (Review)**： 專案管理員會進行代碼審查。若需進一步調整，直接在原主題分支推送新 commit 即可，PR 頁面會自動更新。

## 5. 開發注意事項
* **保持同步**： 提交 PR 前，請務必先同步主專案最新的 `main` 代碼，並在本機解決所有「合併衝突」，確保 PR 是可以順利 Auto-merge 的狀態。
* **嚴禁敏感資訊**： 絕對不要將 API Key、密碼、個人隱私或金鑰等敏感資訊寫入代碼！一旦推送到 GitHub，所有歷史紀錄都將公開可見。
* **單一任務原則**： 一個 PR 應只處理一個特定的主題。多個不相關的變更請分開多個 PR 提交，以利快速審查。
