# 部署架構對比與評估指南：全地端雙機架構 vs 雲端主體混合架構

本指南旨在針對工會系統的兩種部署方案進行深度對比分析，提供決策層與技術團隊評估依據：

2. **方案 B：雲端主體混合架構**（參考 [雲端主體部署架構圖.md](file:///c:/Users/TMP-214/Desktop/project/Labor_union/document/地端部屬/雲端主體部署架構圖.md)）

## 一、 兩大架構簡介與定位

```
【方案 A：全地端雙機架構】
[行政人員] ──(專屬網頁)──> [地端 Laptop (FastAPI/Streamlit/Ollama/Watcher)] ──(LAN)──> [地端 NAS (MySQL)]
                          ▲
[LINE Server] ──(Funnel)──┘

【方案 B：雲端主體混合架構】
[行政人員] ──(專屬網頁)──> [雲端 VPS (FastAPI/Streamlit/Agent/Watcher)] ══(Tailscale VPN)══> [地端 NAS (MySQL)]
                          ▲
[LINE Server] ──(Nginx)───┘
```

* **方案 A（全地端雙機架構）**：應用服務（FastAPI、Streamlit、File Watcher、Ollama 本地 LLM）完全部署於地端工作站 Laptop 上，資料庫獨立放置於地端 Synology NAS。外網 LINE Webhook 透過 Tailscale Funnel / 公網隧道引入。
* **方案 B（雲端主體混合架構）**：應用服務移至雲端虛擬主機 (Cloud VPS)，資料庫 (MySQL) 依然保留在地端 Synology NAS 中。雲端與地端之間透過加密 VPN 隧道 (Tailscale / WireGuard) 串接。

## 二、 多維度比較分析

### 1. 💰 成本考量 (Cost)

| 評估項目 | 方案 A：全地端雙機架構 | 方案 B：雲端主體混合架構 |
| :--- | :--- | :--- |
| **初期硬體成本** | **高**。需配備中高階 Laptop/工作站（執行 Docker 及 Ollama 本地 LLM 需要較高記憶體及 GPU/CPU 算力）。 | **低**。現有辦公電腦即可作為終端，雲端主機採訂閱制，無需採購高規 Laptop。 |
| **定期營運費用** | **極低**。幾乎無雲端伺服器租金（僅需基本網域與 Tailscale 免費/入門方案）。 | **中/低**。需支付雲端 VPS 租金（約 300~800 元/月）與 API 調用費（若使用雲端 LLM）。 |
| **電力與硬體損耗** | 需 24/7 不間斷開機，工作站有硬體老化、電池膨脹或停電中斷風險。 | 雲端伺服器由廠商維護電力與硬體，地端僅需維持 NAS 運行。 |
| **隱性維護成本** | **較高**。地端工作站遭人為誤關機、網路中斷時需人工排查。 | **低**。雲端廠商提供高可用性 (SLA) 與自動重啟機制。 |

### 2. 🛠️ 部署難度 (Deployment Complexity)

| 評估項目 | 方案 A：全地端雙機架構 | 方案 B：雲端主體混合架構 |
| :--- | :--- | :--- |
| **環境建置門檻** | **中等**。需在地端 Laptop 安裝 Docker Desktop、Ollama 本地模型、設定開機自啟與服務防呆。 | **中高**。需在雲端 VPS 設定 Docker、Nginx、網域 SSL，並建置 Tailscale VPN 跨網隧道。 |
| **AI 模型部署** | **較難**。需在地端環境下載並調整 Ollama 模型參數（如 `Qwen2.5-Coder`），需顧慮顯卡/記憶體相容性。 | **極簡**。可直接串接雲端 LLM API（如 OpenAI / Gemini），或於 VPS 上部署大算力模型。 |
| **網路穿透與轉發** | 需要公網隧道工具（Tailscale Funnel / Cloudflare Tunnel）將流量穿透至地端 Laptop。 | 雲端 VPS 本身具備固定公網 IP，直接搭配 Nginx 做標準 SSL 反向代理即可。 |

### 3. 🔧 系統維護與穩定度 (Operations & Maintenance)

| 評估項目 | 方案 A：全地端雙機架構 | 方案 B：雲端主體混合架構 |
| :--- | :--- | :--- |
| **服務可用性 (Availability)** | **較受限**。若辦公室停電、斷網或 Laptop 被誤關機，LINE Bot 與管理後台將立即中斷。 | **極高 (99.9%)**。雲端主機 24 小時在線，LINE Bot Webhook 穩定運作不中斷。 |
| **遠端存取與維護** | 若地端網路斷開，工程團隊無法遠端連入修復，需至現場處理。 | 工程團隊可隨時透過 SSH 遠端連入雲端 VPS 進行修補與升級。 |
| **災難復原與備份** | 依賴 Laptop 與 NAS 的手動/自動 Cron 備份。Laptop 損壞時重建需重新配置環境。 | 雲端 VPS 可一鍵建立快照 (Snapshot) 鏡像，地端 NAS 使用 Hyper Backup 自動加密異地備份。 |

### 4. 🔒 資安、資料主權與隱私 (Security & Data Governance)

| 評估項目 | 方案 A：全地端雙機架構 | 方案 B：雲端主體混合架構 |
| :--- | :--- | :--- |
| **資料庫隱私與主權** | **極佳**。資料庫與應用完全存放在工會內部，無任何數據流經外網伺服器。 | **極佳**。**MySQL 資料庫依然保留在地端 NAS**，雲端僅處理無狀態 (Stateless) 邏輯。 |
| **對外暴露邊界** | 需開放隧道將外網 Webhook 引入地端區域網路，需防範隧道被作為跳板。 | 地端 NAS 完全不對外網開放 Port 3306，經由 VPN 安全閘道前置防護，資料庫隱形於公網。 |
| **AI 數據隱私** | **100% 本地化**。採用 Ollama 本地模型，對話與個資絕不傳送至第三方雲端。 | 若使用雲端 API 需去標籤化；若 VPS 部署私有模型則同樣具備高隱私性。 |

## 四、 決策建議與選型指引 (Decision Framework)

### 💡 建議選擇「方案 A（全地端雙機架構）」的情境：
1. **預算導向 (零月費需求)**：工會**完全不希望產生任何月租型雲端費用**。
2. **極致隱私要求**：工會規章明確規定「所有 AI 對話與客戶個資，連運算過程都嚴禁經過任何雲端 VPS 或 API」。
3. **具備在地 IT 維護人員**：辦公室有人員能在工作站異常或斷電時進行重啟與基本維護。

### 🚀 建議選擇「方案 B（雲端主體混合架構）」的情境（推薦最佳解）：
1. **追求高可用性與穩定度**：LINE 官方帳號需要 24/7 穩定回覆客戶，不能因辦公室停電或網路中斷而漏接訊息。
2. **無高規硬體採購計畫**：辦公室無高配備 Laptop/工作站，希望以低月費直接取得高品質雲端運算資源。
3. **彈性遠端維運**：系統未來有持續升級與維護需求，希望開發與維運團隊能隨時遠端進行維護與問題排查。
4. **兼顧資料主權**：**MySQL 資料庫依然鎖在地端 NAS**，既享受雲端服務的高穩定度，又保有絕對的資料安全與主權。
