# LINE Bot 系統操作與架構 SOP (標準作業程序)

這份文件旨在協助團隊成員快速理解目前 LINE Bot 的三大核心功能區塊：**Webhook 接收與判斷**、**LIFF 註冊與綁定表單**，以及 **圖文選單 (Rich Menu) 切換機制**。系統為了方便後續維護，已將文字與設定全面無碼化 (抽取為 JSON 設定檔)。

---

## 🌟 白話文圖解

為了讓所有人都能輕鬆看懂，這裡用情境與選項的方式說明：當用戶點擊或輸入什麼，系統背後到底會做哪些事？

### 📱 情境一：一般用戶 (剛加入好友或預設狀態)
當客戶一加入 LINE 官方帳號，下方會顯示藍色底的**「一般用戶選單」**。上面有兩個按鈕：

1. **點擊【尋找專員】按鈕**：
   *   **用戶動作**：點擊後，系統會自動幫客戶在聊天室輸入「尋找專員」四個字。
   *   **系統反應**：這些文字會被送到後台的 AI 模組 (RAG) 或是自動回覆系統，根據工會的知識庫給予客戶適當的回應與協助。

2. **點擊【查詢訂單 (LIFF)】按鈕**：
   *   **用戶動作**：點擊後，聊天室會自動發送「查詢訂單」。
   *   **系統反應**：機器人會回傳一條專屬網址。點開後，會跳出「工會登記入口網頁」(LIFF)，畫面提供**兩個選項**：
       *   👉 **選項 A【已填過政府表單】**：適合舊客戶。點擊後進入 `bind.html`，只需輸入「姓名」與「電話」並送出。
       *   👉 **選項 B【未填過政府表單】**：適合新客戶。點擊後進入 `register.html`，這是一份包含 60 題的完整產婦服務登記表單 (取代原本的 Beclass 表單)，需詳細填寫基本資料、新生兒狀況、月子餐喜忌等。
   *   **系統背後的超大工程**：
       1. **比對身分**：系統拿著填寫的姓名跟電話，去工會資料庫瘋狂比對。
       2. **新客戶建檔**：如果是新客戶，系統會幫他建檔，並自動產生一筆「預設的服務訂單」。
       3. **舊客戶綁定與防呆**：如果找到舊資料，系統會把這筆資料跟他的 LINE 綁在一起。如果發現這筆資料「已經被別人的 LINE 綁走過了」，網頁會跳出警告視窗詢問：「這個人已經綁過別的 LINE 囉，你要重新綁定嗎？」。若選擇重新綁定，系統會將需求寫入 `config/rebind_requests.json` 暫存檔，等待服務人員人工確認後才會覆蓋。
       4. **完美收尾**：綁定成功後，網頁會打綠色勾勾，LINE 聊天室會主動推播 `clients.case_no` 作為案件編號；尚未核發時會通知客戶等待行政核對。

### 👩‍🍼 情境二：月嫂專屬身分切換與專屬選單
月嫂有自己專屬的紅色底**「月嫂專區」**選單。但剛加入時看不到，必須透過「通關密語」來切換。

1. **輸入通關密語【我是月嫂】**：
   *   **用戶動作**：在 LINE 聊天室直接打字輸入 `我是月嫂`。
   *   **系統反應**：系統發現密語，立刻向 LINE 總部發送指令，把該名用戶的選單**強制切換成紅色的「月嫂專區」**。機器人會回傳「身分驗證成功」。

2. **月嫂專區的兩個專屬按鈕**：
   *   👉 **按鈕【訂單查詢】**：點擊後，自動發送「訂單查詢」文字，系統會去撈取指派給該月嫂的訂單狀態並回傳。
   *   👉 **按鈕【班表查詢】**：點擊後，自動發送「班表查詢」文字，系統會撈取該月嫂近期的排班表。

3. **輸入還原密語【esc】**：
   *   **用戶動作**：在聊天室輸入 `esc`。
   *   **系統反應**：系統會解除該月嫂的專屬綁定，把下方的選單**變回原本藍色的「一般用戶選單」**，並回傳「已為您解除月嫂身分」。這主要是為了方便內部營運團隊或工程師，在開發測試時可以隨時來回切換身分。

## 1. Webhook 自動回覆機制
**程式位置**：`line/line_bot.py` 的 LINE Router，由 `api/main.py` 掛載
**運作邏輯**：LINE 官方帳號收到訊息後，會將用戶對話事件傳送到此路由，系統根據「關鍵字」進行攔截與處理。

### 精準動作細節：
*   **非同步推播機制**：
    為了避免處理超時，Webhook 收到訊息後**不會直接卡住等待發送**，而是將準備要回覆的訊息內容、該用戶的 `user_id` 與 `scheduled_at` 寫入 `line_tasks`。Webhook commit 後喚醒 `line/worker.py`；Worker 會在新任務通知或最近排程時間到達時執行，不再每 2 秒輪詢。
*   **攔截關鍵字：「我是月嫂」**
    *   **動作**：系統會向 LINE API 發送 `POST` 請求，強制將該用戶的介面切換為「月嫂專屬圖文選單」。
    *   **回覆**：寫入資料庫推播「身分驗證成功」之訊息 (文字由 `webhook_replies.json` 定義)。若系統尚未設定月嫂選單，則提示「系統尚未設定」。
*   **攔截關鍵字：「esc」**
    *   **動作**：向 LINE API 發送 `DELETE` 請求，解除該用戶的個人化圖文選單，強制讓用戶退回「系統預設的一般用戶選單」。
    *   **回覆**：寫入資料庫推播「已為您解除月嫂身分」之提示。
*   **攔截關鍵字：「查詢訂單」或「綁定」**
    *   **動作**：系統從環境變數或資料庫抓取 `LINE_LIFF_ID`，自動組裝出專屬的 `https://liff.line.me/{liff_id}` 網址。
    *   **回覆**：推播包含 LIFF 連結的引導文案，讓用戶點擊填寫。
*   **一般聊天 (未攔截之文字)**
    *   會被送入 ChromaDB 與 RAG (檢索增強生成) 模組，由 AI 進行智慧回答。

## 2. LIFF 表單綁定與派單功能
**前端位置**：`api/static/register.html`
**後端位置**：`api/main.py` -> `@app.post("/api/line/register")`

### 精準動作細節：
*   **前端動態渲染 (入口頁 `gateway.html`)**：
    用戶在 LINE 點擊連結後會先進入入口網頁，網頁會透過 LIFF SDK 默默取得該用戶的 LINE User ID。畫面提供兩條路線：
    1. **已填過政府表單 (`bind.html`)**：供舊客戶綁定 LINE 帳號，僅需填寫姓名與行動電話。
    2. **未填過政府表單 (`register.html`)**：供新客戶建檔，需填寫 60 題完整表單（取代原先外部的 Beclass 表單）。
*   **資料提交與比對**：
    無論客戶走哪一條路線，送出資料後，後端會進入 `clients` 與 `orders` 資料表進行建檔與比對：
    1.  **查無資料 (新用戶)**：系統直接在 `clients` 新增該用戶，綁定 LINE ID，並「自動產生一筆初始預設訂單 (Order)」，最後寫入 `beclass_records` 以相容舊有匯入資料系統。
    2.  **找到資料 (舊客戶首次綁定)**：若 `line_user_id` 欄位為空，則直接更新填入 LINE ID。
    3.  **找到資料 (帳號被占用)**：若該客戶資料已經綁定過別的 LINE ID，系統會回傳狀態 `confirm_rebind`，前端會跳出對話方塊詢問用戶「是否重新綁定」。確認後，會將申請送至後台等待管理員人工審核，不會直接覆蓋資料庫。
*   **最終回傳**：
    綁定完成後，系統會讀取 `clients.case_no`，回傳給網頁顯示，並透過 Webhook 機制推播案件編號給用戶。

## 3. 雙層圖文選單 (Rich Menu) 系統
**建置腳本**：`line/setup_rich_menus.py`
**運作邏輯**：系統支援同時存在兩套選單，方便區分一般客戶與服務人員。

### 精準動作細節：
*   **選單 1：一般用戶預設選單 (Default Menu)**
    *   包含「查詢訂單」(直接開啟 LIFF) 與「尋找專員」(發送文字) 兩個功能。
    *   腳本執行時，會將此選單透過 `POST /v2/bot/user/all/richmenu/{richMenuId}` 設定為整台機器人的**預設選單**，所有剛加入好友的人都會看到這個。
*   **選單 2：月嫂專屬選單 (Caregiver Menu)**
    *   包含「訂單查詢」與「班表查詢」兩個純文字按鈕。
    *   腳本執行時，**不會**將它設為預設，而是將取得的選單 ID 儲存進資料庫的 `settings` 表中 (`caregiver_rich_menu_id`)。
    *   這套選單必須依賴第一階段 Webhook 攔截到「我是月嫂」指令時，才會單獨賦予給發話的那個用戶。
*   **注意**：選單的圖案是由程式利用 Python 的 ImageDraw 動態畫出來並上傳的，不需要人工準備圖檔。

## 4. 備註：JSON 動態設定檔 (無碼化管理)
為了讓非工程背景的營運團隊也能維護 LINE Bot，所有顯示文字、對話、顏色皆獨立放置於 `config/` 資料夾，**存檔後即刻生效（不需重啟伺服器）**。

1.  **`config/webhook_replies.json`**：
    *   管理 Webhook 發送的自動對話。
    *   **特殊設計**：支援字串替換變數，修改文案時務必保留 `{bind_url}`、`{name}`、`{case_no}` 等大括號參數，後端程式會自動將真實在線資料塞入這些位置。
2.  **`config/liff_settings.json`**：
    *   控制 LIFF 網頁的外觀與顯示標題。
    *   **特殊設計**：可直接修改 `theme_colors.primary` 與 `background_gradient` 替換工會的網頁主題色（例如櫻花粉、工會藍）。
3.  **`config/line_menu.json`**：
    *   **控制那兩張圖文選單上的按鈕文字 (`text`)、動作指令與顏色。
    *   **⚠️ 唯一例外**：因為圖文選單是註冊在 LINE 的雲端伺服器上，所以修改此檔案後，**必須手動執行 `uv run python line/setup_rich_menus.py`**，讓系統重新畫一張新選單傳給 LINE，改動才會生效。

4.  **`db/form_templates.json` (內部管理設定檔)**：
    *   這個檔案用來定義「合約書」或「月嫂行前通知單」要呈現哪些欄位（例如：新生兒數量、飲食喜忌、樓層費等）。
    *   它是未來自動產出表單與 PDF 合約的重要架構，修改時需確保 `db_key` 與資料庫欄位名稱對應。
